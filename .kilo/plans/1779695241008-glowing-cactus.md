# Fix: Nav Toggle Not Working Outside Hero Section

## Root Causes

### Bug 1: `nav:injected` fired twice → duplicate toggle listeners
`shared-nav.js` dispatches `nav:injected` for **both** `inject('site-nav', ...)` and `inject('site-footer', ...)`. Each dispatch triggers `nav.js`'s `initNav()`, which calls `toggle.addEventListener('click', ...)` again. Two identical listeners on the toggle cause: first listener opens menu, second listener immediately closes it in the same click cycle → menu appears broken.

### Bug 2: `buildNav()` destroys `#nav-backdrop`
`content.js:buildNav()` replaces `.nav__menu` innerHTML with only `<a>` links, removing `<div id="nav-backdrop">` used for mobile menu click-to-close.

## Fix Plan

### 1. `js/shared-nav.js` — Stop double-firing `nav:injected`
- Dispatch `nav:injected` only for the nav injection, not the footer injection.
- Change the nav `inject()` to dispatch `nav:injected`; the footer `inject()` should dispatch a different event or nothing at all (nav.js doesn't need it).

### 2. `js/nav.js` — Guard against duplicate initialization
- Add a `_initialized` flag so `initNav()` only runs once, even if `nav:injected` fires multiple times from any source.

### 3. `js/content.js` — Preserve `#nav-backdrop` in `buildNav()`
- When rebuilding `.nav__menu` links, include the `<div id="nav-backdrop">` element so the mobile overlay backdrop is not lost.

## Files to Change
- `js/shared-nav.js:18` — scope nav:injected dispatch to nav only
- `js/nav.js:17-23` — add `_initialized` guard to `initNav()`
- `js/content.js:133` — include `#nav-backdrop` in rebuilt innerHTML
